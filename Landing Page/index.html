The Wedge

SBOMs are now a legal requirement.

Most teams are not ready. The deadlines are already set. The penalties are real.

Executive Order 14028

Effective now (US Federal Procurement)

Any software sold to the US federal government must include an SBOM. CISA updated the minimum elements in 2025, adding component hash, license information, tool name, and generation context.

  • CycloneDX or SPDX format required
  • 9 minimum data fields per component
  • Component hashes and license data now mandatory
  • Golden Dome program requires full bills of materials

EU Cyber Resilience Act

Vulnerability reporting: Sept 2026. Full compliance: Dec 2027

The first EU-wide regulation mandating cybersecurity requirements for digital products throughout their entire lifecycle. SBOMs are required for compliance.

  • 24-hour vulnerability reporting to ENISA starts Sept 2026
  • SBOMs effectively required before Sept 2026 (reporting prerequisite)
  • Fines up to 15M EUR or 2.5% of global turnover
  • CE marking required for software sold in EU market

FAQ

Questions you're probably asking.

What exactly is an SBOM and why do I need one?
An SBOM (Software Bill of Materials) is a machine-readable inventory of every component in your software, including open-source dependencies, their versions, licenses, and where they came from. You need one because EO 14028 requires it for US federal procurement, the EU Cyber Resilience Act requires it for anything sold in the EU market, and your enterprise customers are already asking for it in security reviews. If you sell software to anyone with a procurement process, you will need to produce one. The question is whether you cobble it together from scripts or generate it with one click.
Does PackrAI read my source code?
PackrAI reads your commit messages, lock files, Dockerfile, and API surface to produce release artifacts. It does not read, store, or transmit your application source code. Repo clones are ephemeral and destroyed after generation completes. Your code never leaves the generation worker.
Which SBOM formats does PackrAI support?
CycloneDX JSON is the default. It's the format CISA and the CRA both recognize and the one most procurement teams expect. SPDX JSON is also available on request. Both include all CISA 2025 minimum elements: component name, version, supplier, unique identifiers (purl, cpe), component hash, license, dependency relationships, SBOM author, generation timestamp, tool name, and generation context.
How accurate are the SBOMs?
PackrAI generates SBOMs from lock files (package-lock.json, requirements.txt, go.sum, Cargo.lock), which capture both direct and transitive dependencies with exact version pins. This is the most reliable method for most projects. We flag the confidence level on each SBOM and note where transitive resolution may be incomplete (e.g., projects without lock files). Binary analysis for compiled languages is on the roadmap for Phase 4.
Does it work with private repos?
Yes, on the Pro and Team plans. You connect your GitHub account via OAuth, authorize access to the repos you choose, and PackrAI generates from there. We only request read access to the repos you select. Your OAuth token is encrypted at rest and never shared.
What languages and ecosystems are supported?
Phase 1 supports Node.js (package-lock.json, yarn.lock, pnpm-lock.yaml), Python (requirements.txt, Pipfile.lock), and Go (go.sum). Phase 2 adds Rust (Cargo.lock), Ruby (Gemfile.lock), and PHP (composer.lock). Phase 3 adds Java (Maven, Gradle) and .NET. If your stack is not listed, join the waitlist and tell us what you need. We build based on demand.
Can I integrate PackrAI into my CI/CD pipeline?
Yes, on the Team plan. You configure a webhook on your GitHub repo, and when a tag is pushed, PackrAI automatically generates the full artifact set and publishes the GitHub Release draft. We also provide a GitHub Action for more granular control. API access is available for custom integrations.
How is this different from Syft or release-drafter?
Syft generates SBOMs and only SBOMs. release-drafter generates changelogs and only changelogs. PackrAI generates the complete release artifact set: changelog, SBOM, Dockerfile audit, deploy bundle, docs delta, and GitHub Release. You would need to wire together four or five separate tools and custom scripts to approximate what PackrAI does with one click. That wiring is exactly what most teams never get around to doing.
What about the EU Cyber Resilience Act deadlines?
The vulnerability reporting obligation starts September 11, 2026. You must report actively exploited vulnerabilities to ENISA within 24 hours. Full compliance (SBOMs, secure-by-design, CE marking, technical documentation) is required by December 11, 2027. Practically, you need SBOMs before September 2026 because accurate component-level visibility is a prerequisite for meeting the 24-hour reporting requirement. PackrAI gets you there.
Is my SBOM stored? Can I retrieve past SBOMs?
Yes. Every SBOM PackrAI generates is stored and retrievable by version, date, and format. On the Team plan, you get a full SBOM version history with comparison views between releases. This matters when a vulnerability is disclosed and you need to quickly identify which of your past releases are affected.
